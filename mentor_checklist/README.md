# Mentor Checklist — Fintech Hackathon (15-Minute Session)

## Purpose
A structured checklist to quickly evaluate early-stage teams for **clarity, feasibility, and execution readiness** within a 15-minute mentoring session.

---

# 0. Pre-Session Setup (Mental)
- [ ] Keep strict time control (15 minutes total)
- [ ] Focus on **clarity, data, feasibility, MVP**
- [ ] Avoid giving solutions — identify weaknesses
- [ ] Prepare to interrupt vague answers

---

# 1. Pitch (0:00–1:30)
### Goal: Extract core idea quickly

- [ ] Problem is stated clearly in one sentence
- [ ] User segment is explicitly defined
- [ ] Solution is understandable without jargon
- [ ] Hackathon deliverable is mentioned

### Red Flags
- [ ] Vague users (“everyone”, “people”)
- [ ] Buzzword-heavy (“AI-powered fintech platform”)
- [ ] No concrete outcome

---

# 2. Problem & User Validation (1:30–4:00)
### Goal: Ensure real problem exists

- [ ] Specific user persona identified
- [ ] Real-world scenario described step-by-step
- [ ] Current solutions acknowledged
- [ ] Pain point is meaningful (not trivial)

### Red Flags
- [ ] Hypothetical problem
- [ ] No awareness of existing solutions
- [ ] Weak or indirect value

---

# 3. Value Proposition & Differentiation (4:00–6:30)
### Goal: Determine competitive edge

- [ ] Clear reason why users would switch
- [ ] Competitors identified (at least 2–3)
- [ ] Differentiation is concrete
- [ ] Improvement is significant (not marginal)

### Red Flags
- [ ] “Better UX” as sole differentiator
- [ ] No competitor knowledge
- [ ] Feature disguised as product

---

# 4. Data & Technical Feasibility (6:30–9:30)
### Goal: Stress-test realism

- [ ] Required data clearly identified
- [ ] Data source exists and is accessible
- [ ] APIs or datasets are known
- [ ] System architecture roughly outlined

### If AI is used:
- [ ] Input/output clearly defined
- [ ] Justification for AI vs rules
- [ ] Evaluation method described

### Red Flags
- [ ] “We will get data later”
- [ ] No API access
- [ ] AI used without necessity
- [ ] Overly complex architecture

---

# 5. Scope & Execution Plan (9:30–11:30)
### Goal: Ensure realistic MVP

- [ ] Clear MVP defined
- [ ] Demo scenario described
- [ ] Features prioritized
- [ ] Non-essential features identified for removal

### Red Flags
- [ ] Full product instead of MVP
- [ ] No demo plan
- [ ] Over-scoped (mobile + backend + AI + blockchain)

---

# 6. Risk Identification (11:30–13:00)
### Goal: Surface critical uncertainties

- [ ] Team identifies main technical risk
- [ ] Key assumption acknowledged
- [ ] Hardest component identified

### Red Flags
- [ ] “No major risks”
- [ ] Overconfidence
- [ ] Ignoring dependencies

---

# 7. Security & Compliance (13:00–14:30)
### Goal: Basic fintech awareness

- [ ] Sensitive data identified
- [ ] Storage/security approach mentioned
- [ ] GDPR awareness
- [ ] KYC/AML considerations (if relevant)

### Red Flags
- [ ] No consideration of data protection
- [ ] Storing sensitive data casually
- [ ] Ignoring regulatory constraints

---

# 8. Final Feedback (14:30–15:00)
### Deliver concise, actionable guidance

## Strengths
- [ ] Identify 1–2 strong aspects

## Critical Weaknesses
- [ ] Identify up to 3 major risks

## Immediate Next Steps (within 48 hours)
- [ ] Validate data source
- [ ] Reduce scope
- [ ] Build minimal demo component

# Intervention Prompts (Use When Needed)

- [ ] “Who exactly is the user?”
- [ ] “Where does the data come from?”
- [ ] “What will you actually demo?”
- [ ] “This sounds like a feature—what is the product?”
- [ ] “What happens if your assumption is wrong?”

---

# Common Failure Patterns

- [ ] No real data access
- [ ] Overuse of AI without justification
- [ ] Overly ambitious scope
- [ ] No monetization thinking
- [ ] Ignoring compliance

---

# Expected Outcome of Session

- [ ] Sharper problem definition
- [ ] Clear MVP scope
- [ ] Identified key risks
- [ ] Concrete short-term plan
